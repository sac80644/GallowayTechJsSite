.NET - Why isn't this documented?

It took me a little while to figure this one out. I have used URL Rewrites and Redirects before but based off some querystring after the domain name, not a redirect from root. If you do a search on URL Rewrites you will find the usage of either Regular Expressions, Wildcards or Exact Match for the match url. The example below is for the web.config file and the expression has a caret ^ which denotes the beginning of a string and dollar sign $ for the end of the string. So it is matching the empty string. In other words, just the domain name (i.e. http://yourdomain.com/). Change the Redirect to Rewrite and the browser URL will not be updated so the transfer is transparent. The rewrite may break your site links though, such as css. So test it locally.

URL Redirect


    <system.webServer>
        <rewrite>
            <rules>
                <rule name="Root Redirect" stopProcessing="true">
                    <match url="^$" />
                    <action type="Redirect" url="/mySubFolder" />
                </rule>
            </rules>
        </rewrite>
    </system.webServer>

Spread Operator

I came across something like this not too long ago:

var foo = new Set(...args);
And I thought, what the heck is that?!? So I looked it up. I found the ... is referred to as the spread operator, or on the Mozilla site as Spread Syntax. This looks exactly like Rest Syntax but not the same. Anyways, today I used it to push an array of object into another array of object like such:

let firstArray: MyObject[] = [{key: "value1"}, {key: "value2"}, {key: "value3"}];
let secondArray: MyObject[] = [{key: "value4"}, {key: "value5"}, {key: "value6"}];
firstArray.push(...secondArray);
And voila. The secondArray's objects are expanded out and pushed into the firstArray. This can also be done with Array.concat() or some other methods but I like this one...

Webpack - Setting up Hot Module Replacement

So I have gone from enjoying, to being frustrated and back to enjoying using Webpack. It can do a lot but is difficult to get completely set up. Luckily for me the project I am on during the day already has it up and running. The one thing that I really wanted to see was Hot Module Replacement, and that would prove to be difficult.

Webpack does have some nice online guides to help you get started but the guides are mostly just giving you the super ideal and simple scenarios. I went through most of the guides and the Hot Module Replacement guide was towards the very end. You can check it out here (https://webpack.js.org/guides/hot-module-replacement) but you need to have followed steps up to that point to get your project working. Or you can get my demo project that I scaled down after going through all the guides just to highlight HMR: GitHub

HMR Versus Live Reloading

So if you go to the webpack-hmr-demo GitHub repository mentioned above, you will find that it is the simplest implementation of Webpack HMR that you can set up. If you go to the webpack.config.js file, you will find a setting (hot: true) under the devServer section. If you comment this out or change it to false, then hot module replacement will be turned off. By default though, Live Reloading will be enabled. If you run the project and in Chrome open dev tools, under console you will see [WDS] Live Reloading enabled. If you leave the hot set to true you will also see [WDS] Hot Module Replacement enabled. So what is the difference?

Live reloading will basically re-build and re-load your site for you after you make any modification and save. This can be a nice time saver, but if your build time is long then saving you a click or two doesn't help that much. Hot module replacement can help to speed up development in that it only rebuilds the module that you made changes to and reloads only that module, not the entire site. You do have to build your site using modules in such a way to take advantage of this feature but if you build it right then it is pretty cool. Download my HMR Demo, get it running and read Webpack's HMR guide and it should make more sense to you than it did the first time for me.
