The How, What and Why Decisions in Building This Site

So I have written many sites using ASP .NET and to be honest, I am bored. I even created a site using the "new", at the time of this writing, .NET Core 2.2 and Angular 6.0 project. I did enjoy working with the lean .NET Core framework but it was just still more of the same.
It reminds me of when I went backpacking through Europe for the first time with my brother and friend. When we landed and got over the jet lag, we did the usual tourist things in London and then started moving onto eleven different countries. We wanted to go to places where people did not speak English. We wanted something different. So, that is no dig on London, nor am I dissing .NET. But like I said, I got bored and wanted something different and new.

Artitechtual Decisions

I mentioned my on, off and on again relationship with Javascript in this site's intro, and I recently began working on a project that is built on a .NET Core Razor Pages framework, but 90% of the work is done in Typescript. And that of course means needing to know Javascript. Which I am very happy about to be able to put in some solid time usins JS as I feel we never really got to know eachother.

So Typescript during the day at work, and Javascript at night on this site. I then took the idea of components that I learned building some Angular sites. So each component consists of one .html file, one .css and one .js. The .js imports the content and becomes an encapsulated component exposing both content and functionality. This is where the lightbulb lit up in my head on callbacks which I use in the navbar component to expose events.

Lastly, at my day job we use Webpack so that is what I use on this site for bundling, plugins, build, webserver, loaders, and my personal fav - hot module replacement. I did get my feet wet with gulp and I like how straight forward it is to get going, but Webpack is what I stuck with.

So to wrap up, this site is a Javascript driven, Bootstrap responsive, component based Single Page App (SPA). This site was developed using VS Code (love it!), npm, Webpack and git to GitHub for source control. Oh, and also continuous integration between GitHub and my GoDaddy site using SemaphoreCI.

Jimmy Bogard - Verticls Slice Architecture.

So a few months ago at the office I was asked to watch this video lecture by Jimmy Bogard. There are a bunch out there, just do a search Jimmy Bogard Verticls Slice Architecture. Or his own site has a short article.

After watching and getting a good understanding we began a new effort to update and consolidate a large amout of site pages into dashboards. Instead of the usual n tier architecture we broke up our site into "feature modules." Each feature module was a completely separate project so nothing was shared. With this approach we were able to have smaller releases. This meant quicker turn around from our QA team, less complexity and developers could work without stepping on eachother.

Since we are not currently performing any commands, just queries, I have not been able to fully see the pros and cons of this approach. But for what we have done so far I think the part I like the most is that as a developer I get to write more of my owne code. Instead of using building blocks that someone else wrote to cobble together a new feature, I get to make more decisions and contribute more. This is a WET vs. DRY approach and it takes some getting used to. There is a tendency to make every component from a feature module a generic shared component. We are early on in this development effort and I can already see some components growing too complext, but we will figure that out soon enough.

In short, I recommend at least watching a presentation, reading about Verticle Slices on Jimmy's site or another's. In his demo he uses MediatR. Read up on that too. It is not complex, it is very simple and liberating.
